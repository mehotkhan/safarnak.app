# AGENTS.md

Guidance for Codex and other coding agents working in this repository.

## Project Overview

Safarnak is a single-root full-stack, offline-first travel app. It is not a Yarn workspace.

- Client: Expo React Native, Expo Router, React 19, NativeWind v4, Apollo Client, Redux Toolkit.
- Server: Cloudflare Worker with GraphQL Yoga, D1, KV, R2, Vectorize, Durable Objects, Queues, Workflows, and Workers AI.
- Shared GraphQL: `graphql/schema.graphql` and `graphql/queries/*.graphql`.
- Shared database: `database/schema.ts` is the source of truth for Drizzle tables, with separate adapters for server and client.
- Generated client API: `api/types.ts` and `api/hooks.ts` are generated by GraphQL Codegen.
- Current package version and scripts come from `package.json`; README badges and examples can lag behind code.
- Current quality baseline: `yarn lint` and `./node_modules/.bin/tsc --noEmit` are expected to pass before/after refactors.

## Important Directories

- `app/`: Expo Router pages and route groups.
- `src/ui/`: client UI components, hooks, state, theme, layout, forms, maps, utilities.
- `api/`: client Apollo setup, cache storage, utilities, generated hooks/types.
- `graphql/`: shared schema, operations, generated worker declarations, schema loader.
- `database/`: shared Drizzle schema plus server/client adapters.
- `worker/`: Cloudflare Worker entry, GraphQL resolvers, domains, storage, Durable Objects, utilities.
- `locales/`: English and Persian translations.
- `assets/`: images, fonts, app assets.
- `migrations/`: server D1 migrations.

## Current App Shape

The active Expo tabs are defined in `app/(app)/_layout.tsx`:

- `(home)`: home feed, post details, new post, inspirations.
- `(explore)`: search/discovery, locations, places, users, shareable trips, map comparison.
- `(trips)`: trips, trip details/edit/map, locations/places creation, shareable trip list.
- `(inbox)`: inbox, conversations, message detail.
- `(me)`: profile, account completion, saved/history/studio, payments, subscription, settings.
- `compose`: hidden from the tab bar and used as a compose modal/sheet route.

Auth routes live under `app/(auth)` with `welcome`, `login`, and `register`.

## Critical Rules

- When docs conflict, trust live source files first: `package.json`, `tsconfig.json`, `metro.config.js`, `wrangler.jsonc`, and the actual `app/`, `src/ui/`, `worker/`, `database/`, and `graphql/` trees.
- Do not add `"type": "module"` to `package.json`; CommonJS config files rely on the current package type.
- Keep `eslint.config.mjs` as the ESLint flat config.
- Never manually edit generated files: `api/hooks.ts` and `api/types.ts`.
- Run `yarn codegen` after changing `graphql/schema.graphql` or files in `graphql/queries/`.
- Keep server/client boundaries strict:
  - `worker/` is server-only.
  - `api/` is client-only.
  - `graphql/` is shared.
  - `database/schema.ts` is shared; use adapter-specific imports.
- Use `@database/server` in worker code and `@database/client` in client code.
- Prefer NativeWind `className` styling over inline React Native styles.
- Run `yarn db:migrate` before testing local Worker APIs that depend on D1 schema changes.
- Worker entry is configured as `worker/index.ts`; do not change Worker config entry points casually.
- Keep secrets and tokens out of client code and committed files.
- Do not add new placeholder resolvers, mock-only production paths, or TODO-only implementations for GraphQL fields.
- Keep `worker/version.ts` synchronized with `package.json` when touching Worker release/version behavior.

## Imports And Aliases

Use path aliases for cross-directory and ownership-boundary imports. Current aliases are defined in `tsconfig.json` and Metro config:

- `@/*` -> repo root
- `@ui/*` -> `src/ui/*`
- `@graphql/*` -> `graphql/*`
- `@database/*` -> `database/*`
- `@worker/*` -> `worker/*`
- `@api`, `@api/*` -> `api`
- `@state`, `@state/*` -> `src/ui/state`
- `@hooks`, `@hooks/*` -> `src/ui/hooks`
- `@constants`, `@constants/*` -> `constants`
- `@utils`, `@utils/*` -> `src/ui/utils`
- `@locales`, `@locales/*` -> `locales`
- `@assets`, `@assets/*` -> `assets`
- `@migrations`, `@migrations/*` -> `migrations`

Examples:

```ts
import { useMeQuery } from '@api';
import { getServerDB, users } from '@database/server';
import { getLocalDB, cachedTrips } from '@database/client';
import { useAppDispatch } from '@state/hooks';
import { generateRandomName } from '@utils/nameGenerator';
```

The current codebase still has many local sibling relative imports, especially in `worker/`, `database/`, and barrel files. Do not churn those mechanically. When refactoring a touched module, prefer moving deep cross-boundary imports to aliases while keeping simple same-folder imports acceptable.

## GraphQL Workflow

When adding or changing GraphQL behavior:

1. Update `graphql/schema.graphql` if the schema changes.
2. Add or update operations in `graphql/queries/*.graphql`.
3. Run `yarn codegen`.
4. Implement or update worker resolvers under `worker/`.
5. Use generated hooks from `@api` in client code.

Client components should import generated hooks from `@api`; do not import raw `.graphql` operations directly into UI code.

## Product And Data Domains

- Social/feed uses normalized `feed_events`, `feed_preferences`, `follow_edges`, `close_friends`, and feed subscriptions such as `feedNewEvents`.
- Search has lexical operations (`search`, `searchSuggest`) and semantic search (`searchSemantic`) backed by Workers AI embeddings and Vectorize.
- Trending prefers KV-backed top lists and uses the `TrendingRollup` Durable Object for decay/compaction.
- Embedding jobs are sent through `EMBED_QUEUE`; the default Workers AI embedding model is `@cf/baai/bge-m3` and the Vectorize index is configured for 1024 dimensions.
- Media such as avatars is stored in R2; token/session and verification data use KV.
- Subscriptions use the `SubscriptionPool` Durable Object.

## Refactoring Operating Mode

The long-term goal is a fast, clean, production-ready codebase for global release. Work in small vertical slices that keep the app shippable.

- Preserve behavior first, then improve structure. Avoid big-bang rewrites.
- Keep lint and TypeScript clean. If a change touches GraphQL, run `yarn codegen`; if it touches DB schema, run migration generation/apply commands as appropriate.
- Prefer deleting dead legacy paths over maintaining compatibility shims, but only after confirming schema, operations, routes, and UI references are gone.
- Convert repeated parsing/mapping logic into shared mappers only when at least two active paths need it.
- Do not expand large screens/hooks further; extract cohesive hooks/components when touching files over roughly 300 lines.
- Keep offline-first behavior intact: Apollo cache, Drizzle cache, AsyncStorage queue, and local SQLite migrations must continue to work without network.
- User-facing changes need English and Persian translation coverage.

## Production Readiness Priorities

1. Security and privacy:
   - New auth/session work should move sensitive client material toward `expo-secure-store` or platform keychain/keystore patterns. Existing token/private-key storage in AsyncStorage is a known risk, not a pattern to copy.
   - Avoid logging tokens, private keys, signatures, PII, full GraphQL variables, or AI prompts containing user-sensitive content.
   - In production Worker paths, prefer masked client errors plus structured server logs; do not leak stack traces or internal failure details.
   - Release builds must use real release signing, not debug signing.

2. Data contract correctness:
   - Treat `graphql/schema.graphql`, `graphql/queries/*.graphql`, worker resolvers, `database/schema.ts`, and generated `api/*` files as one contract.
   - Keep hosted trips as `Trip` with `isHosted`; do not reintroduce `Tour` tables/types unless explicitly requested.
   - Consolidate Apollo-to-Drizzle entity transforms; the current transform logic exists in both `api/cache-storage.ts` and `database/client.ts`.
   - Add database indexes when adding feed/search/list queries that filter or order by new columns.

3. Performance:
   - Main feed, explore, trips, maps, and trip-detail flows are hot paths.
   - Avoid N+1 resolver patterns for feed/search/detail expansion; batch by IDs or join where practical.
   - Prefer list virtualization improvements on large lists. If adding a dependency for this, justify it against bundle size.
   - Lazy-load heavy features such as maps, subscriptions/WebSocket-only code, and AI-heavy flows where possible.
   - Prefer `expo-image` or the established image component strategy for image-heavy screens.

4. Worker architecture:
   - `worker/index.ts` is still a large entrypoint that wires schema, GraphQL server, HTTP assets, queue, scheduled jobs, and exports. Future refactors should split this into server/router/handlers/queue/scheduled modules.
   - Empty `worker/server`, `worker/domains/*`, and `worker/storage` directories exist; do not assume their documented refactor is complete.
   - Keep resolvers thin; move reusable business logic into `worker/utilities/*` or future domain services.
   - Use Cloudflare bindings from `Env`; do not hardcode service credentials or API keys.

5. Client architecture:
   - Keep `app/` route files focused on screen orchestration.
   - Put reusable UI into `src/ui/display`, `src/ui/forms`, `src/ui/feedback`, `src/ui/cards`, `src/ui/layout`, `src/ui/modals`, or `src/ui/components` as appropriate.
   - Put reusable logic into `src/ui/hooks` or `src/ui/utils`; avoid duplicating screen-local data transforms.
   - Prefer consistent `LoadingState`, `ErrorState`, and `EmptyState` components over one-off state UI.
   - Keep route strings aligned with the current groups: `(home)`, `(explore)`, `(trips)`, `(inbox)`, `(me)`, and `compose`.

## Known Hotspots

Be extra careful when editing these files because they are central, large, or currently carry production risk:

- `app/_layout.tsx`: app bootstrap, providers, font/cache readiness, offline queue processing.
- `api/client.ts`: GraphQL URL resolution, auth link, WebSocket link, Apollo cache persistence.
- `api/cache-storage.ts`: Apollo persistence plus structured Drizzle sync.
- `database/client.ts`: Expo SQLite initialization, migrations, stats, legacy sync helper.
- `database/schema.ts`: shared server/client schema and migration source.
- `worker/index.ts`: Worker entrypoint, Yoga setup, auth context, static assets, queue/scheduled handlers.
- `worker/mutations/createTrip.ts`: trip creation, workflow kickoff, feed/search/trending/vector side effects.
- `worker/queries/getFeed.ts`: feed pagination and entity expansion; currently sensitive to N+1 growth.
- `src/ui/hooks/useAuth.ts` and `src/ui/hooks/crypto.ts`: biometric auth and private key handling.
- Large screens such as `app/(app)/(trips)/[id]/index.tsx`, `app/(app)/(explore)/index.tsx`, and `app/(app)/(trips)/new.tsx`.

## Database Workflow

- Server D1 and client SQLite tables are coordinated through `database/schema.ts`.
- Server adapter: `database/server.ts`, imported as `@database/server`.
- Client adapter: `database/client.ts`, imported as `@database/client`.
- Server migrations are generated from the shared schema and stored in `migrations/`.

For server table changes:

1. Update `database/schema.ts`.
2. Run `yarn db:generate`.
3. Run `yarn db:migrate` for local D1.
4. Update GraphQL schema/operations and run `yarn codegen` if the change is exposed to the client.

For client cached table changes:

1. Update `database/schema.ts`.
2. Include sync metadata fields where appropriate (`cachedAt`, `lastSyncAt`, `pending`, `deletedAt`).
3. Confirm `api/cache-storage.ts` sync behavior if Apollo cache data should populate the table.

## Client Patterns

- Use functional React components and typed props.
- Use generated Apollo hooks directly from `@api`.
- The Apollo cache is persisted through `DrizzleCacheStorage`; do not create wrapper hooks unless there is a real product need.
- Redux state lives under `src/ui/state`.
- Keep Redux state lean; rely on Apollo cache and local Drizzle tables for data where appropriate.
- Add or update translations in both `locales/en/translation.json` and `locales/fa/translation.json` for user-facing text.
- Persian/Farsi translations exist, but RTL layout is currently not forced.

## Worker Patterns

- Worker config is `wrangler.jsonc`; local Worker dev runs on port `8787`.
- Worker bindings include `DB`, `KV`, `R2`, `VECTORIZE`, `AI`, `EMBED_QUEUE`, `SUBSCRIPTION_POOL`, `TRENDING_ROLLUP`, `TRIP_CREATION_WORKFLOW`, and `TRIP_UPDATE_WORKFLOW`.
- Validate resolver inputs and fail fast with clear errors.
- Use Drizzle ORM for D1 access; avoid raw SQL unless there is a strong reason.
- Keep auth/session/token logic in worker utilities.
- Use Cloudflare bindings from the resolver context/environment rather than hardcoded credentials.
- Keep Worker imports server-safe; do not import React Native, Expo, Apollo client code, or UI modules.

## Styling

- Prefer NativeWind utility classes via `className`.
- Use existing theme, layout, form, feedback, and display components under `src/ui/` before adding new primitives.
- Keep UI consistent with the app: dense, practical travel/social/product surfaces, not marketing pages unless explicitly requested.
- Avoid inline styles except for dynamic values or APIs that require style objects.

## TypeScript And Quality

- Type component props, function parameters, and return values for public or non-obvious functions.
- `any`, assertions, and `@ts-expect-error` are allowed when they are the pragmatic option; add an explanation for suppressions.
- Prefer existing local utilities and patterns over new abstractions.
- Before finishing meaningful code changes, run the narrowest useful checks:
  - `yarn lint`
  - `./node_modules/.bin/tsc --noEmit`
  - `yarn codegen` for GraphQL changes
  - `yarn db:generate` / `yarn db:migrate` for database changes

## Debugging Notes

- Metro issues: run `yarn clean`, then restart Expo.
- Worker API testing: check `http://localhost:8787/graphql`.
- D1 issues: inspect `.wrangler/state/v3/d1/` and re-run `yarn db:migrate`.
- Type or generated hook issues: run `yarn codegen`.
- Alias resolution issues: check both `tsconfig.json` and `metro.config.js`.

## Git And Release Notes

- Version source of truth is `package.json`.
- CI release behavior follows conventional commit prefixes:
  - `feat:` or `feature:` -> minor bump.
  - `fix:` or `bugfix:` -> patch bump.
  - other prefixes usually build without version bump.
- Do not revert unrelated user changes in the working tree.
